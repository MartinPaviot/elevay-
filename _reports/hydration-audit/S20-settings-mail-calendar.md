# S20 — settings-mail-calendar (`/settings/mail-calendar`) — audit d'hydratation

**Verdict global : H2 (partiel).** The page is genuinely wired: it fetches GET /api/settings/mail-calendar on mount (real, tenant+user-scoped reads of authAccounts, connectedMailboxes, and tenants.settings) and persists sync preferences via PUT → updateTenantSettings (merges into tenants.settings, tenant-scoped). Loading skeleton, empty state, and error state are all handled, and the three sync-preference controls load their stored values and round-trip on save. The main fidelity gaps are two route-level placeholders: lastEmailSyncAt/lastCalSyncAt are hardcoded null (TODO) so the per-account "Email sync" timestamp never shows a real value, and calendarConnected is hardcoded true for every OAuth account rather than reflecting actual calendar state.

## Pires défauts

1. Per-account 'Email sync {timeAgo}' is a placeholder: route.ts:145 hardcodes lastEmailSyncAt=null (TODO Phase B), so the UI at page.tsx:445-450 never shows a real last-sync time and always falls back to the literal 'active'.
2. calendarConnected is hardcoded true for every OAuth account at route.ts:141, so the 'Calendar connected' badge (page.tsx:451-456) does not reflect actual calendar-connection state — it is asserted, not loaded.
3. Save does not reflect back from the server: handleSavePreferences (page.tsx:270-290) never re-fetches after PUT, relying on optimistic local state, so any server-side normalization (e.g. sanitizeDomains dedup/trim at route.ts:241) is not surfaced until a manual reload.
