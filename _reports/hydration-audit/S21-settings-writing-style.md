# S21 — settings-writing-style (`/settings/writing-style`) — audit d'hydratation

**Verdict global : H2 (partiel).** A genuinely well-wired settings surface: every control (about-me, role, sign-off, scheduling link, the editable prompt with server-supplied default, tone preset, per-audience variants) loads its current value from real owner-scoped user_preferences (resource inbox) and saves round-trip back. Reset uses the server defaultPrompt, the derive proposal is a real polled async pipeline, and the routing preview is genuinely tenant-scoped (contacts lookup by tenantId). The one meaningful gap is load error handling: a failed writing-style GET is swallowed by .catch(()=>{}) leaving style=null, which pins the page on a permanent spinner with no written error state.

## Pires défauts

1. Load error is silently swallowed: the writing-style fetch rejects on non-ok (page.tsx:60) but .catch(()=>{}) (page.tsx:72) eats it; setLoading(false) runs while style stays null, so `loading \|\| !style` (page.tsx:210) keeps the page on a permanent spinner with no written error/retry state (vs Home's per-lane written empty/error degradation).
2. All loads are tied to a single combined spinner gate (page.tsx:210); a partial failure of any one of the three parallel fetches has no independent degradation — the voice/derive fetches fall back to defaults silently but a writing-style failure blocks the whole page.
3. No surfaced error for derive/voice load failures (page:61-62 fall back to {} defaults silently), so a backend outage on tone/proposal looks identical to 'no data' — minor silent-stale risk.
