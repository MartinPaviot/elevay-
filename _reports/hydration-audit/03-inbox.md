# 03 — inbox (`/inbox`) — audit d'hydratation

**Verdict global : H2 (partiel).** The /inbox page is a strong, near-reference-bar implementation: every data-bearing surface (conversation list, folder/split/mailbox counts, reading pane, bundles, catch-up banner, header subtitle) is wired to real data that is rigorously scoped to the signed-in user's own mailbox(es) within the tenant (getInboxScope filters connected_mailboxes by tenantId AND userId; scopeConversationRows narrows the read-model; the detail route 404s on a cross-user key). Primary surfaces degrade independently with skeletons, a retryable error EmptyState, and written per-lane/per-split empty copy that self-hides. The only fidelity gaps are two secondary surfaces that fetch real data but swallow load errors into a misleading empty state — the Outbound table (catch+console.error, no retry) and the Capture review drawer (no loading state, silent error → renders nothing).

Entrée : `app/apps/web/src/app/(dashboard)/inbox/page.tsx`.

## Pires défauts

1. Outbound table swallows fetch failures into a silent empty table — `.catch(console.error)` with no error/retry state, so a 500 looks like 'no sent mail' (app/apps/web/src/app/(dashboard)/inbox/_outbound-table.tsx:89-90), unlike the conversation list which has listError+Retry
2. Capture review drawer has no loading state and silently swallows fetch errors (renders nothing on failure), so a failing approval queue is invisible (app/apps/web/src/app/(dashboard)/inbox/_capture-review.tsx:27-38)
3. Capture drawer tenant scoping is unverified at this layer — /api/inbox/captures was not inspected; the element relies entirely on that route enforcing tenant/user scope (app/apps/web/src/app/(dashboard)/inbox/_capture-review.tsx:29)
