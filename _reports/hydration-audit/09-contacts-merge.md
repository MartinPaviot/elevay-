# 09 — contacts-merge (`/contacts/merge`) — audit d'hydratation

**Verdict global : H2 (partiel).** The page is largely faithful: both data sources (GET /api/contacts/merge for auto-detected duplicate groups, GET /api/contacts for curated selections) are real and properly tenant-scoped (eq(tenantId) + deletedAt guards), with loading, written empty state, and independent error toasts handled, matching the Home-page bar. The merge POST is also tenant-verified before any write. The two defects are silent-partial-data issues, not unwired or broken elements: curated mode fetches only the first 50 contacts (no pageSize), so preselected ids beyond row 50 are silently dropped and can trigger a false 'need 2 valid contacts' error; and the auto-mode candidate avatar always passes domain=null so it never shows the real company logo. Worst meaningful state is H2.

## Pires défauts

1. Curated mode loads only the first 50 contacts: loadCurated calls fetch('/api/contacts') with no page/pageSize (page.tsx:99), defaulting to pageSize 50 (route.ts:38); any preselected id beyond row 50 fails byId.get (page.tsx:106-109) and is silently dropped, producing a false 'Need at least 2 valid contacts to merge' error (page.tsx:110-114). Use ?ids=/idsOnly or fetch by the explicit ids instead.
2. Auto-mode candidate avatar is never the real company logo: CompanyLogo is hardcoded domain={null} (page.tsx:301) because GET /api/contacts/merge does not return companyDomain (route.ts:50-62), so it always falls back to initials despite candidates having a companyId.
3. Curated form silently swallows a failed /api/contacts fetch into an error toast but leaves no inline empty/error region in the curated branch — a non-ok response yields a toast then an empty page with no retry affordance (page.tsx:100-103,188-195).
