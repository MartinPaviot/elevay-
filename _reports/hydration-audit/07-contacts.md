# 07 — contacts (`/contacts`) — audit d'hydratation

**Verdict global : H2 (partiel).** The /contacts page is near-reference quality on data hydration: every data-bearing element traces to GET /api/contacts (or /api/import/history), both rigorously tenant-scoped via getAuthContext + eq(tenantId), with soft-delete filtering, a full TableSkeleton loading state, per-cell '—' empties, two distinct written empty states, and server-side filters/counts spanning ALL contacts. The single meaningful gap is error degradation: the list fetch swallows failures to console.warn with no error banner, so a 500/network failure renders the fresh-tenant 'No contacts yet' empty state — a failure looks identical to an empty tenant. Hence H2 rather than H1; no mock/unwired/leaking elements found.

## Pires défauts

1. List-fetch failure is swallowed (console.warn, no error UI) at page.tsx:248-250 → a failed/500 GET /api/contacts leaves contacts=[] and renders the 'No contacts yet' import CTA (page.tsx:1339-1348), masking an error as an empty tenant. No independent per-lane error degradation like the Home reference.
2. Company-detail join query (route.ts:286-289) filters only by id ANY(), not by tenantId — safe today because ids come from this tenant's contacts, but it relies on that invariant rather than an explicit eq(companies.tenantId) guard.
3. Import History 'completed/partial' status badge has no empty/error written state (panel just self-hides at page.tsx:1633); a fetch failure silently yields [] with no signal, though impact is minor.
