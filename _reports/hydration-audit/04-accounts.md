# 04 — accounts (`/accounts`) — audit d'hydratation

**Verdict global : H1 (fidèle).** The /accounts page is broadly faithful (H1): every data-bearing element traces to a real, tenant-scoped data source. The primary list, counts, facets and per-row intelligence all go through /api/accounts and sibling routes that scope on eq(companies.tenantId, authCtx.tenantId), with a TableSkeleton loading state and three distinct, written empty states (no accounts / no search match / no active-filter match) plus per-facet try/catch so sub-queries degrade independently. The one systemic weakness is error handling on the main list fetch: on a non-200 or thrown error fetchAccounts only console.warns and returns, leaving the page stuck in skeleton (or showing stale rows) with no user-facing error banner — so a 500 is invisible rather than surfaced.

## Pires défauts

1. Main list fetch swallows errors with no UI: app/apps/web/src/app/(dashboard)/accounts/page.tsx:729 (`if (!res.ok) return;`) and :744 (catch → console.warn) — a 500 from /api/accounts leaves a permanent skeleton or stale list with no error state, unlike Home's independent-degradation bar.
2. Warm-paths column (page.tsx:877 /api/warm-paths) tenant scoping not verified in this pass — companyIds are client-supplied; if the route doesn't re-scope to authCtx.tenantId it is a potential cross-tenant read (flagged tenantScoped=unknown).
3. refetchLoadedAccounts (page.tsx:762) silently `break`s on a failed page and shows whatever partial set loaded, with no toast/error — post-mutation refreshes can present a truncated list as if complete.
