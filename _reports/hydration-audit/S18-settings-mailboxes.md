# S18 — settings-mailboxes (`/settings/mailboxes`) — audit d'hydratation

**Verdict global : H2 (partiel).** /settings/mailboxes is a pure server redirect to /settings/mail-calendar; the effective page is well-wired. Connected accounts and all three sync-preference controls (contact-creation mode, backsync range, ignored domains) load their current values from real tenant-scoped data (connectedMailboxes scoped by tenantId+userId; syncPreferences from tenants.settings scoped by tenant id) and save round-trips through a merge-and-invalidate PUT with server-side validation; loading skeleton, empty state and error state are all handled. Two partials keep it from full fidelity: the per-account 'Email sync … ago' timestamp is fed by lastEmailSyncAt which the API hardcodes to null (an unwired Phase-B TODO that silently degrades to the word 'active'), and Save only flips a local 'Saved' flag without re-fetching the server-sanitized values.

Entrée : `app/apps/web/src/app/(dashboard)/settings/mailboxes/page.tsx`.

## Pires défauts

1. Per-account last-sync timestamp is never wired: route.ts:145-146 and :172 hardcode lastEmailSyncAt/lastCalSyncAt to null with a 'TODO Phase B' comment, so the freshness label at page.tsx:445-450 silently falls back to the static text 'active' (H2 silent-stale freshness).
2. Save does not reflect back: handleSavePreferences (page.tsx:270-290) sets a local 'Saved' badge but never calls loadData(), so the server-sanitized doNotTrackDomains returned by route.ts:241-256 is not re-synced into the UI (H2).
3. Audited route is a redirect-only stub (page.tsx:3-5); the real data-bearing UI is in mail-calendar/page.tsx — anyone reasoning about /settings/mailboxes by file name alone would find no wiring at the named entry file.
