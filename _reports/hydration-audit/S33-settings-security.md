# S33 — settings-security (`/settings/security`) — audit d'hydratation

**Verdict global : H2 (partiel).** This page is genuinely wired, not a stub. The password form is a write-only credential control (passwords are never pre-loaded by design); its POST to /api/account/password verifies the current password via bcrypt against the real per-user hash, persists the new hash, and revokes sessions — faithful round-trip with toast error + saving-state handling. The MFA card's status badge, recovery-codes-remaining counter, and conditional enable/disable UI all reflect REAL per-user persisted state loaded from GET /api/account/mfa (getMfaStatus over the user_mfa_secrets row), and every mutation (POST/PUT/DELETE) reflects back via refresh(). Scoping is per signed-in auth user (authCtx.userId), the correct scope for a personal security setting. The only real defect is a missing loading/error state on the MFA card: status starts null (nothing renders, no skeleton) and refresh() swallows fetch failures in an empty catch, leaving the card silently blank rather than showing an error.

Entrée : `app/apps/web/src/app/(dashboard)/settings/security/page.tsx`.
