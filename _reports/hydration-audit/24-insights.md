# 24 — insights (`/insights`) — audit d'hydratation

**Verdict global : H2 (partiel).** The /insights page is wired to three real, tenant-scoped API routes (pipeline, alerts, briefs), all gated by getAuthContext() and filtered with eq(deals.tenantId)/eq(activities.tenantId)/eq(coachingInsights.tenantId). Data origin is faithful (H1-grade): KPIs and funnel come from real open-deal aggregation, alerts from stalled deals + SLA + coaching insights, and briefs are LLM-generated but grounded in real deal/activity/context-graph rows. ## Pires défauts

1. No error UI anywhere: page.tsx:104 `.catch(console.error)` swallows all three fetch failures, leaving pipeline/alerts/briefs null so the page renders KPIs as 0/$0K/— and hides Alerts & Briefs — a 500 looks identical to an empty tenant (H2 silent, violates Home's independent error degradation bar).
2. No written empty states: Alerts (page.tsx:198) and Deal Briefs (page.tsx:249) sections simply vanish when empty (totalAlerts===0 / briefs.length===0) instead of showing written empty copy like the Home reference page requires.
3. Single shared loading skeleton (page.tsx:108-125) blocks the whole page on the slowest call — and briefs run synchronous LLM generation per request (deal-briefing.ts:260-271) — so one slow/failed lane stalls all lanes instead of degrading independently.
