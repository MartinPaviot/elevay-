# 08 — contact-detail (`/contacts/[id]`) — audit d'hydratation

**Verdict global : H2 (partiel).** The page is broadly faithful: every data-bearing element is wired to real, tenant-scoped sources (contact, activities, buyer-intent, calls, accounts all filter by authCtx.tenantId; calls use withAuthRLS), the header/details show a load skeleton, and sidebar cards self-hide when empty with written empty states. It misses the H1 bar on independent error degradation: several lanes swallow fetch errors and render an empty/absent state instead of an error state, so a 500 looks like 'no data'. No tenant leaks, no mock/placeholder data, no permanent skeletons — worst meaningful state is H2 (partial: missing error handling / silent degraded display).

## Pires défauts

1. Activities lane masks errors as empty: a failed/500 GET /api/activities is caught silently (page.tsx:402-405 + outer catch page.tsx:444) and the timeline renders the identical 'No activity recorded for this contact.' empty state (page.tsx:529-532) — the user cannot tell a real-empty contact from a backend error.
2. Buyer Intent card has no loading or error state: the fetch is wrapped in a swallow-all try/catch (page.tsx:408-416) with no skeleton; on a 500 from /api/contacts/[id]/buyer-intent the card simply never appears (page.tsx:594), indistinguishable from a genuinely low-signal contact.
3. Associated-companies silently degrade to a raw truncated UUID: when GET /api/accounts/[id] fails the catch skips (page.tsx:436) and the link renders 'cid.slice(0,8)+"..."' (page.tsx:676) instead of the company name or an error — a degraded, unhelpful placeholder shown as if it were real data.
