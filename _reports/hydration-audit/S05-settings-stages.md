# S05 — settings-stages (`/settings/stages`) — audit d'hydratation

**Verdict global : H2 (partiel).** The page is largely faithful: stages load from the real tenant config (tenants.settings.pipelineStages, scoped by eq(tenants.id, authCtx.tenantId)) with a loading skeleton, an error string, and a whole-list PUT that persists and is admin-gated + audit-logged. Core fields (name, description, category) are H1. The defect is partial fidelity on the two newer per-stage controls — aiFillMode and wipLimit: they round-trip only because the JSON is re-read untyped, but the loader's PipelineStage type and DEFAULT_STAGES (route.ts:7-23) omit both fields, so on an unconfigured tenant (or any tenant before first save) these controls always render their hardcoded client defaults rather than stored state. There is no written empty state, though DEFAULT_STAGES makes a truly empty list unreachable.

Entrée : `app/apps/web/src/app/(dashboard)/settings/stages/page.tsx`.

## Pires défauts

1. aiFillMode toggle (page.tsx:203) is not part of the loader contract: route's PipelineStage interface (route.ts:7-12) and DEFAULT_STAGES (route.ts:14-23) omit aiFillMode, so an unconfigured tenant always sees the client default 'suggest' (page.tsx:54) instead of a stored value.
2. wipLimit input (page.tsx:218) has the same gap: omitted from DEFAULT_STAGES / PipelineStage type (route.ts:7-23), so it never reflects a stored value until the user saves at least once; loader contract does not guarantee the field.
3. No written empty state for the stage lists (page.tsx:184,258) — an empty saved list would render blank; masked only by the DEFAULT_STAGES fallback (route.ts:34).
