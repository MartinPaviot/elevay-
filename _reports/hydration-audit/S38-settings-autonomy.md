# S38 — settings-autonomy (`/settings/autonomy`) — audit d'hydratation

**Verdict global : H2 (partiel).** A genuinely well-wired settings page: every control hydrates from a real tenant-scoped GET (autonomyConfig + systemTrustScore + tenant settings.learnedThresholds, all eq(tenantId)) and saves round-trip via PUT followed by a re-fetch, with a server-side trust-floor gate on level raises. The only fidelity gap is error handling — fetchConfig swallows failures silently (catch{} with no error UI), so on a failed/slow GET the guardrail inputs fall back to hardcoded defaults (40/25/5) that are visually indistinguishable from real stored values, and the page has only a loading skeleton, no written empty/error state. Trust card and thresholds gate their render so they degrade cleanly; the guardrail block does not, which is the H2.

## Pires défauts

1. Silent fetch error: fetchConfig catch{} swallows the failure with no error UI (page.tsx:93-94), so a failed GET leaves the form showing hardcoded guardrail defaults indistinguishable from stored data — silent-stale risk.
2. Guardrail inputs default to 40/25/5 (page.tsx:84-86) which render the same whether the GET succeeded with those values or failed entirely — no way for the user to tell defaults from real config on error.
3. No written empty/error state for the whole page beyond a loading skeleton (page.tsx:136-142); below the reference Home bar where each lane degrades to its own written empty state.
