# 02 — chat (`/chat`) — audit d'hydratation

**Verdict global : H2 (partiel).** The /chat page is a conversational AI surface, not a KPI dashboard, and its core data lanes are faithfully wired: thread history and live streamed turns come from real tenant/user-scoped sources (GET /api/chat/threads/[id] scoped by userId, useChat POST /api/chat), with a full-page load spinner, a written empty hero, a StreamingSkeleton thinking state, and an independent error banner with Retry. Greeting/firstName and action cards are likewise real tenant-scoped data that degrade cleanly. The only meaningful weakness is the starter suggestions lane (H2): it is personalized off real tenant onboarding fields (role/primaryChallenge) but the phrases are hardcoded canned strings with no loading state, so the static fallback visibly swaps to fetched content, and suggestion fetch failures are swallowed silently.

Entrée : `app/apps/web/src/app/(dashboard)/chat/page.tsx`.

## Pires défauts

1. Starter suggestions have no loading state and swap from a hardcoded 4-item fallback to fetched content after /api/chat/suggestions resolves (app/apps/web/src/app/(dashboard)/chat/page.tsx:519-527), and the suggestion strings themselves are canned phrases keyed off two enum fields rather than live pipeline data (app/apps/web/src/app/api/chat/suggestions/route.ts:16-53).
2. Suggestions + firstName fetch failure is silently swallowed with no user-visible degradation signal (app/apps/web/src/app/(dashboard)/chat/page.tsx:144-146); the user just sees generic fallbacks with no indication personalization failed.
3. Thread GET/POST authorize by userId only, not tenantId (app/apps/web/src/app/api/chat/threads/[id]/route.ts:30,105); safe today because users belong to one tenant, but it is a weaker scope than the eq(tenantId) bar used elsewhere and would leak across tenants if a user were ever multi-tenant.
