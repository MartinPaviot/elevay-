# 06 — account-brain (`/accounts/[id]/brain`) — audit d'hydratation

**Verdict global : H2 (partiel).** The brain page is well-built: a single client fetch to GET /api/brain/[companyId] with auth-derived tenantId (never client-supplied), full loading (DetailPageSkeleton), a rendered error fallback, and a written empty state per section. Company-core, contacts, deals, activities, and meetings are correctly scoped by BOTH companyId AND tenantId and are faithfully hydrated (H1). The systemic defect: three whole sections — Knowledge, Graph facts, and Memories — are filtered ONLY by tenantId, not by companyId, so they render the entire tenant's data (capped) identically on every account's brain page, misrepresenting them as company-specific. The champion badge inherits the same un-company-scoped contextGraphEdges. No tenant leak (everything stays within the caller's tenant) and no broken sources, so worst meaningful state is H2 (partial/wrong-scope), not H5.

Entrée : `app/apps/web/src/app/(dashboard)/accounts/[id]/brain/page.tsx`.

## Pires défauts

1. Knowledge section is NOT company-scoped: query filters only eq(knowledgeEntries.tenantId) with no companyId predicate (get-brain.ts:205), so /accounts/[id]/brain shows the whole tenant's knowledge (capped 20) on every account — count and list are misleading (page.tsx:456)
2. Memories section is NOT company-scoped: chatMemories filtered only by eq(tenantId) (get-brain.ts:225), rendering the tenant-wide chat-memory store (capped 25) identically on every account's brain (page.tsx:504)
3. Graph facts AND the contact champion badge derive from contextGraphEdges filtered only by eq(tenantId) (get-brain.ts:216) with no companyId/company-node filter — shows all tenant edges and can mislabel champions across companies (page.tsx:309, page.tsx:479)
