# S31 — settings-agent-memory (`/settings/agent-memory`) — audit d'hydratation

**Verdict global : H2 (partiel).** This is a read-only memory-inspection panel, not a control/toggle settings page — there are no editable controls, only a display of the tenant's aggregated memory plus an Export JSON button. Every data-bearing element is wired to genuinely tenant-scoped data: getTenantSettings (eq tenants.id), getTenantKnowledge (eq knowledgeEntries.tenantId), and trustEvents (eq trustEvents.tenantId), all keyed off getAuthContext().tenantId. Data fidelity is faithful (H1) across the board; the only meaningful defect is degradation: a fetch failure is swallowed (if(!res.ok) return) leaving snapshot null with no error or empty state, so the page renders nearly blank silently (H2).

## Pires défauts

1. Silent fetch-error handling: page.tsx:80 `if (!res.ok) return;` swallows non-200 responses, leaving snapshot=null and loading=false → page renders only the 'Generated …' header line with no error message and no retry (page.tsx:76-89).
2. No empty/error state written for the body: page.tsx:143-221 render nothing when snapshot is null after a failed load; unlike the Home reference bar, lanes do not degrade to a written empty state.
3. Per-category empty within a successful snapshot is unmarked: groups with zero items return null (page.tsx:146) so a tenant with sparse memory sees missing sections silently rather than an explicit 'nothing learned yet' line.
