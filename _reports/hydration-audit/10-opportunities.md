# 10 — opportunities (`/opportunities`) — audit d'hydratation

**Verdict global : H2 (partiel).** The page is largely faithful: every data-bearing lane (board, table, forecast, KPIs, risk badges, stalled count) is wired to real tenant-scoped data via /api/opportunities, /api/pipeline/analytics and /api/forecast — all of which filter by tenantId (forecast even uses withAuthRLS) and exclude soft-deletes. Loading skeletons and written empty states are present for the board, table, and forecast. The systemic gap vs the Home reference bar is error handling: all four client fetches (deals, analytics, forecast, accounts/contacts) swallow failures into console.warn with no written 'couldn't load' state, so a 500 on /api/opportunities renders as an empty pipeline and a 500 on analytics silently removes the KPI strip — stale/empty conflation rather than visible degradation. No tenant-leak or mock/placeholder data found.

## Pires défauts

1. Deals fetch failure is indistinguishable from an empty pipeline: fetchDeals catches errors to console.warn and still runs setLoading(false), so a 500 from /api/opportunities renders the full empty board/table with no error message (page.tsx:281-291).
2. KPI analytics strip silently vanishes on failure: the entire 6-card strip is gated on `analytics &&` (page.tsx:1468) and fetchAnalytics swallows errors (page.tsx:276-279), so an analytics 500 removes the KPIs with no notice or skeleton — no independent written error state like the Home reference bar.
3. Create-modal Account/Contact pickers degrade silently: /api/accounts and /api/contacts fetch errors are swallowed (page.tsx:293-301), leaving empty dropdowns with no 'couldn't load' message; their tenant-scoping was not verifiable from this page.
