# S34 — settings-privacy (`/settings/privacy`) — audit d'hydratation

**Verdict global : H2 (partiel).** The page is mostly faithfully wired: it loads a single tenant-scoped ComplianceData blob from GET /api/settings/compliance (backed by getTenantSettings(authCtx.tenantId)), and the Data Visibility selector both reads its current value from that blob and persists via PUT /api/settings/workspace with optimistic reflect-back. Sub-processor DPA badges are read-only but faithfully reflect stored per-tenant status. The main defects are non-fatal: the Data Region badge is driven by a build-time env var (NEXT_PUBLIC_GDPR_REGION), not tenant config, and the compliance fetch swallows all errors and non-OK responses silently, so a failed/unauth load shows every control at its hardcoded default (visibility "everyone", all DPAs "not_started") with no error state.

Entrée : `app/apps/web/src/app/(dashboard)/settings/privacy/page.tsx`.

## Pires défauts

1. Data region badge reads a build-time public env var NEXT_PUBLIC_GDPR_REGION instead of the tenant's real DB region, so it cannot reflect actual per-tenant data residency and is identical for every tenant (page.tsx:182-184, 254-256).
2. Compliance fetch swallows all errors and ignores non-OK responses (catch {} 'endpoint may not exist yet'), leaving compliance=null so every control silently shows its hardcoded default (visibility 'everyone', all DPAs 'not_started') with no error/empty state surfaced (page.tsx:80-92).
3. Visibility selector only renders Everyone/Private; a tenant whose stored defaultDataVisibility is 'team' (a valid stored value) shows no active option, so the persisted setting is invisible/unreflected (page.tsx:52-70, 278).
