# S27 — settings-recording (`/settings/recording`) — audit d'hydratation

**Verdict global : H1 (fidèle).** This page is faithfully data-hydrated. Every control (auto-record toggle, bot name, branding policy, opt-out reason, primary domain, domain aliases) loads its current value from real tenant-scoped persisted config (tenants.settings JSON, scoped by eq(tenants.id, authCtx.tenantId)) and saves round-trip via PUT /api/settings/workspace with server-side validation and a settings:write permission gate. The 'Not configured' notetaker badge reflects the real env-driven Recall capability. The only soft spots are presentation, not fidelity: there is no per-field loading skeleton (the form briefly shows hardcoded defaults before the GET resolves), and the /api/features fetch error is swallowed silently.

## Pires défauts

1. No loading state: the form mounts with hardcoded useState defaults (enabled=true, botName='Elevay Notetaker', policy='branded') and only overwrites them after the async GET resolves, so a user can briefly see/interact with optimistic defaults before real values arrive (page.tsx:43-53, 64-77) — H2-flavored, not blocking.
2. Silent error on /api/features: the recallai capability fetch swallows failures with .catch(()=>{}) and falls back to notetakerOn=true, so a failed/500 features call wrongly shows the notetaker as configured (page.tsx:57-62).
3. After a successful save the page does NOT re-fetch; it relies on local state matching the persisted payload. Since PUT normalizes values (lowercasing domains, capping aliases at 10, trimming bot name), the displayed value can drift from what was actually stored until a manual reload (page.tsx:101-104).
